# 数据清洗策略分析：噪声内容过滤与 Role 标签体系扩展

## 1. 问题现状

当前文档中存在两类典型的低价值内容：

### 1.1 批量图片占位符

来源：MinerU OCR 转换 PDF 时，将图片转为 `![image](url)` 的 markdown 图片链接。
典型表现：连续 10+ 行纯图片链接，无任何文字内容。

```
# 示例（污染与废弃物管理.md 第 103-125 行）
![image](https://cdn-mineru.openxlab.org.cn/result/.../.jpg)
![image](https://cdn-mineru.openxlab.org.cn/result/.../.jpg)
...（连续 12 张图片）
```

这些图片实际上是 PDF 中嵌入的流程图/示意图，OCR 后只保留了 URL，失去了文字信息。
**危害**：被切入 chunk 后，content 几乎只有 URL，对 embedding 检索和 BM25 都是噪声。

### 1.2 低价值尾部内容

来源：产品说明书、学术论文等的固定格式尾部区段。

典型内容：
- **参考文献**：`[1] Streubel A, Stenzinger A...` 纯引用条目
- **标识符号解释**：包装图标配说明，如"体外诊断医疗器械" + 图片
- **企业信息/备案信息**：公司地址、备案号、页码 `第 3 页 共 4 页`
- **版权声明**

**危害**：这些内容对生物安全问答没有价值，但可能会被关键词规则或 LLM 打上某个 role（如 regulation），污染检索结果。


## 2. 清洗应该在哪个阶段做？

### 方案 A：在文档预处理阶段清洗（split_chunk 之前/期间）

**做法**：在 `split_chunk.py` 的 `parse_lines()` 阶段，扩展 `NOISE_PATTERNS` 和新增过滤规则，直接标记或移除这些行。

**优点**：
- 噪声根本不会进入 Block/Chunk，**下游全链路无感**
- 减少 chunk 数量 → 节省 embedding 成本和 Milvus 存储
- 实现简单，正则匹配即可处理图片行和参考文献区段
- **因果清晰**：这些内容在原始文档层面就是没有语义价值的，不依赖于"角色判断"

**缺点**：
- 某些图片行实际上嵌在有价值的段落之间（如"图2.1 清除污染水平"的配图，前后有文字说明），粗暴删除会割裂上下文
- 参考文献的边界不总是明确（有的文档没有"参考文献"标题，直接以 `[1]` 开头）

### 方案 B：在打标阶段过滤（chunk_processor 新增 role）

**做法**：在 `config.py` 的 `roles` 中新增一个 `junk`（或 `other`/`noise`）标签，让规则/LLM 将低价值 chunk 打上此标签。检索时过滤掉 `role=junk` 的结果。

**优点**：
- 保留了完整数据，万一某些"看似噪声"的内容有用，可以事后恢复
- LLM 打标可以处理更微妙的语义判断（比如区分"有配文字的图片"和"纯图片堆砌"）

**缺点**：
- 噪声 chunk 仍然会消耗 embedding 计算和 Milvus 存储
- 增加了 role 打标的复杂度（LLM 可能在 junk 和其他 role 之间犹豫不决）
- **本质上不合适**：role 体系是用来描述内容的"知识角色"（SOP、法规、应急...），而不是描述"是否有价值"。junk 不是角色，是数据质量属性

### 方案 C：**两阶段组合**（推荐）

| 阶段 | 处理内容 | 策略 |
|------|---------|------|
| **预处理阶段** (split_chunk) | 确定性噪声 | 正则规则硬过滤 |
| **打标阶段** (chunk_processor) | 模糊噪声 | 新增 `junk` 标签 + 检索过滤 |

**理由**：

1. **确定性噪声应该尽早消灭**。连续纯图片行、参考文献区段、企业备案信息——这些用简单规则就能识别，没有"误杀"风险，不应该浪费下游资源。

2. **模糊噪声交给打标兜底**。对于那些"可能有用也可能没用"的灰色地带（比如只有一两行文字 + 几张图片的 chunk，或者"标识的解释"这种边界内容），让 chunk_processor 的规则打分 + LLM 来判断，打上 `junk` 标签，在检索时过滤。

3. **`junk` 标签的定位不是"角色"，而是"质量标记"**。可以考虑不放在 `roles` 里，而是作为独立字段（如 `is_junk: bool`），这样不干扰 role 打分逻辑。或者简单起见先放在 roles 里，但在检索时始终过滤。


## 3. 具体实施方案

### 3.1 预处理阶段：扩展 split_chunk.py

需要新增的过滤规则：

#### (a) 图片行过滤

```python
# 新增 ParsedLine 属性
is_image: bool  # 是否为纯图片链接行

# 判断逻辑
def is_image_line(line: str) -> bool:
    """判断是否为纯 markdown 图片链接行"""
    stripped = line.strip()
    return bool(re.match(r'^!\[.*?\]\(.*?\)\s*$', stripped))
```

处理策略：**不是简单删除，而是标记**。后续在 `_assemble_text()` 中：
- 如果图片行前后有文字说明（如"图2.1 xxx"），保留图片说明文字，删除 URL
- 如果是连续纯图片行（>=3行），整块删除

#### (b) 参考文献/尾部区段截断

```python
# 尾部区段匹配模式
TAIL_SECTION_PATTERNS = [
    r'^#+\s*【?参考文献】?',         # # 参考文献 / # 【参考文献】
    r'^#+\s*【?基本信息】?',         # # 基本信息
    r'^#+\s*【?标识的解释】?',       # # 标识的解释
    r'^\[?\d+\]\s*\w+.*\d{4}',     # [1] Author...2020 (参考文献条目)
]

# 页码模式（已有但可加强）
PAGE_NUMBER_PATTERN = r'^第\s*\d+\s*页\s*共\s*\d+\s*页$'
```

处理策略：
- 检测到"参考文献"标题后，后续所有行标记为 `is_tail_section = True`
- 在 Block 切分时，尾部区段不创建 Block

#### (c) 企业信息/产品头尾模板

```python
BOILERPLATE_PATTERNS = [
    r'^产品使用说明书\s*$',
    r'^.+Biotech\s+Co\.,?\s*Ltd\.?\s*$',
    r'^版本号[:：].*$',
    r'^.+备案凭证编号.*$',
    r'^.+说明书核准日期.*$',
    r'^.+生产日期及失效日期.*$',
]
```

### 3.2 打标阶段：chunk_processor.py 新增质量过滤

两种可选方案：

#### 方案一：新增 `junk` role（最小改动）

```python
# config.py
roles: Tuple[str, ...] = (
    "sop", "emergency", "regulation", "directory",
    "knowledge", "equipment", "reagent", "notice",
    "junk",  # 低价值/无意义内容
)

# chunk_processor.py - 新增 ROLE_KEYWORDS
"junk": [
    ("参考文献", 5), ("引用", 2),
    ("备案", 3), ("备案凭证", 4),
    ("说明书核准", 4), ("版本号", 3),
    ("生产企业", 3), ("售后服务", 3),
],
```

在检索侧过滤：
```python
# retriever.py
filter_expr = f'role != "junk"'
```

#### 方案二：独立质量字段（更干净）

```python
# TaggedChunk 新增字段
is_junk: bool = False
junk_reason: str = ""  # 如 "image_only", "reference_section", "boilerplate"

# score_role 之外，新增独立的 score_junk() 方法
def score_junk(self, text: str, chunk: Chunk) -> Tuple[bool, str]:
    """判断 chunk 是否为低价值内容"""
    # 1. 纯图片链接（content 中图片行占比 > 80%）
    # 2. 文本长度过短（< 20 字有效文字）
    # 3. 关键词匹配（参考文献、备案信息等）
    ...
```

**推荐方案一**，因为：
- 与现有 role 体系兼容，不需要修改 Milvus schema
- 检索过滤简单，一个 `role != "junk"` 就行
- 概念上虽然不够纯粹，但实用性好


### 3.3 质量检测规则（用于 chunk_processor 的 junk 判定）

```python
def _is_junk_chunk(text: str) -> Tuple[bool, str]:
    """判断 chunk 内容是否为低价值噪声"""
    lines = text.strip().split('\n')
    non_empty = [l for l in lines if l.strip()]

    # 规则1: 纯图片 chunk（图片行占比 > 80%）
    image_lines = [l for l in non_empty if re.match(r'!\[.*?\]\(', l.strip())]
    if non_empty and len(image_lines) / len(non_empty) > 0.8:
        return True, "image_only"

    # 规则2: 有效文字过少（去掉图片/URL 后 < 20字）
    clean_text = re.sub(r'!\[.*?\]\(.*?\)', '', text)
    clean_text = re.sub(r'https?://\S+', '', clean_text)
    if len(clean_text.strip()) < 20:
        return True, "too_short"

    # 规则3: 参考文献条目（多行 [数字] Author... 格式）
    ref_lines = [l for l in non_empty if re.match(r'^\[?\d+\]', l.strip())]
    if len(ref_lines) >= 2 and len(ref_lines) / len(non_empty) > 0.5:
        return True, "reference_section"

    return False, ""
```


## 4. 预处理 vs 打标阶段的清洗边界总结

```
                    ┌─────────────────────────────┐
                    │       原始 Markdown 文档       │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │  split_chunk.py (预处理)       │
                    │                              │
                    │  ✂ 删除: 连续纯图片行 (>=3)    │
                    │  ✂ 删除: 页码行               │
                    │  ✂ 截断: 参考文献区段           │
                    │  ✂ 删除: 企业模板文本           │
                    │  ♻ 保留: 有文字说明的图片       │
                    │  ♻ 保留: 不确定的灰色内容       │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │  chunk_processor.py (打标)     │
                    │                              │
                    │  🏷 role 打分 (sop/emergency   │
                    │    /regulation/... 不变)       │
                    │  🏷 junk 检测 (新增):          │
                    │    - 图片占比 > 80%            │
                    │    - 有效文字 < 20 字           │
                    │    - 参考文献条目               │
                    │    → 打标为 role="junk"        │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │  retriever.py (检索)           │
                    │                              │
                    │  🚫 过滤: role="junk"          │
                    └─────────────────────────────┘
```


## 5. 核心设计原则

1. **确定性噪声尽早杀、模糊噪声打标延后杀** —— 不在上游做过于激进的删除，也不把所有脏活丢给下游
2. **宁可多保留、不要误删** —— 被误标为 junk 的 chunk 还在数据库里，可以随时调整规则恢复；被预处理删掉的行就需要重新跑文档了
3. **role 体系的语义纯洁性** —— `junk` 不是一种知识角色，它是质量标记。短期为了实用放在 roles 里没问题，长期应该独立为 `quality` 字段
4. **渐进式完善** —— 先实现最高 ROI 的规则（连续图片行删除 + 参考文献截断），观察效果，再逐步补充细粒度规则
